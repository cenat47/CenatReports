# 📊 Report Service  
Сервис генерации отчётов (CSV) на основе данных из базы данных, с поддержкой фоновых задач, авторизации, проверки статуса и скачивания файлов через API. Проект реализован на FastAPI с использованием Celery, Redis, PostgreSQL, Docker, SIEM (ELK Stack) и HSM для криптографии.

---

## 📝 Описание проекта
Report Service — backend-сервис, который автоматически формирует отчёты на основе данных из БД.  
База данных заполняется отдельным скриптом, после чего сервис генерирует отчёты (сейчас доступно 10 типов) в формате CSV.

Генерация вынесена в фоновые задачи Celery, что позволяет обрабатывать тяжёлые операции без блокировки основного приложения. Через API можно запускать создание отчётов, проверять статус задач и скачивать готовые файлы.

В проекте используется многослойная архитектура (Repository, Data Mapper, Service Layer), DTO на Pydantic и авторизация с использованием JWT и refresh-токенов.

## 🔒 Система безопасности

Проект включает многоуровневую защиту данных и противодействие атакам:

### 🔐 Аутентификация и авторизация
- **JWT-токены** с разделением на access (короткий TTL) и refresh (долгий TTL)
- **Ротация refresh-токенов** — при обновлении старый токен удаляется
- **Хранение токенов в httpOnly cookies** — защита от XSS
- **Роли пользователей**: `user`, `manager`, `admin` с разделением прав доступа
- **Двухфакторное подтверждение** изменения ролей через email-код

### 🛡️ Защита от атак
- **Rate Limiting на уровне Nginx** — ограничение скорости запросов (5/мин для `/login`)
- **Автоматическая блокировка IP** — после 5 неудачных попыток входа IP блокируется на 15 минут через Redis
- **Валидация сложности паролей**:
  - Минимум 8 символов
  - Заглавные и строчные буквы
  - Цифры
  - Только ASCII-символы
- **Security Headers** (X-Content-Type-Options, X-Frame-Options, X-XSS-Protection)
- **Защита от SQL-инъекций** через параметризованные запросы SQLAlchemy

### 📋 Аудит и мониторинг
- **Полный аудит-лог** всех критичных операций:
  - Успешные и неудачные попытки входа
  - Блокировки IP
  - Обновление токенов
  - Изменения ролей пользователей
  - Генерация и скачивание отчётов
  - Попытки несанкционированного доступа
- **Запись IP-адресов и User-Agent** для расследования инцидентов
- **Хранение старых и новых значений** при изменениях данных (для отката)

### 🔑 Контроль доступа к данным
- **Изоляция данных пользователей** — пользователь видит только свои отчёты
- **Проверка прав доступа** на уровне API — попытки доступа к чужим данным логируются
- **Каскадное удаление сессий** при удалении пользователя
- **Сохранение аудит-логов** даже после удаления пользователя (для расследований)

### 🧰 Целостность данных
- **Foreign Key constraints** — невозможно создать отчёт для несуществующего пользователя
- **Уникальность email** — защита от дубликатов
- **NOT NULL для критичных полей** — гарантия наличия обязательных данных
- **Типизация на уровне БД** — PostgreSQL отклонит некорректные данные

## 📚 Архитектура
Проект основан на слоях:

### **1. API-слой**
Обрабатывает запросы, получает DTO, вызывает сервисы.

### **2. Service Layer**
Бизнес-логика:  
- запуск задач генерации  
- проверка статусов  
- взаимодействие с репозиториями и Celery

### **3. Repository Layer**
Изолирует работу с БД.  
Используются Data Mappers для преобразования моделей.

### **4. Celery Worker**
Выполняет обработку данных и генерацию CSV-файлов:
- запрос данных из БД  
- обработка (MV-логика)  
- запись результата в файл  
- сохранение пути к файлу

## ✅ Тестирование

В проекте реализованы интеграционные тесты на основе **pytest**, **httpx** и **pytest-asyncio**.  
Тесты поднимают изолированное тестовое окружение (отдельная БД, mock Redis) и проверяют ключевые сценарии:

- Аутентификация:
  - Успешный логин с корректными учётными данными.
  - Ошибка при неверном пароле (401) без выдачи токенов.
- Доступ к защищённым эндпоинтам:
  - Эндпоинт `/admin/status` недоступен без авторизации (401).
- Работа с отчётами:
  - Авторизованный пользователь с ролью `manager` может инициировать генерацию отчёта через `/report/generate`.
  - В тестовой БД создаётся шаблон отчёта `daily_sales`, на основе которого запускается генерация.
- Роли и данные:
  - В тестовой базе данных присутствует пользователь с ролью `manager`, что проверяется отдельным тестом.

Запуск тестов:
```bash
pytest
```

Тесты не требуют поднятого Redis и реальных Celery‑воркеров: внешние зависимости замоканы на уровне фикстур, что делает прогон быстрым и детерминированным.

## 🛠 Запуск проекта

### 1️⃣ Клонировать репозиторий
```bash
git clone https://github.com/cenat47/CenatReports.git
cd CenatReports
```
### 2️⃣ Создать .env по примеру .env.example
### 3️⃣ Запустить docker-compose

## 🚀 Планы доработки (Q1 2026)

### 🔍 SIEM с Correlation ID (ELK Stack)
- **Elasticsearch 8.x + Kibana 8.x** для централизованного логирования
- **Correlation ID** для трейсинга запросов 
- **Filebeat** в Docker для сбора логов контейнеров

- **Цифровая подпись отчетов** (PDF/A3 + XAdES для compliance)
- **Автоматическая ротация ключей** с HSM (ежемесячно)
- **FIPS 140-2 Level 2** совместимость для regulated industries
